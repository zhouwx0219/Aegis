1. Transaction Context

现有 [transaction.py (line 40)](C:/Users/Administrator/Documents/Aegis/code/cast-das-tmp/agent/runtime/transaction.py:40) 可以作为上层生命周期入口，但底层需要真正的原生事务上下文：

TxnId: task_id + attempt_id + generation

Status: ACTIVE / WAITING / COMMITTING / COMMITTED / ABORTING / ABORTED

Phase: explore / refine / commit

Action: OCC / four locking actions

Priority: current score + last update epoch

RS/WS/HRS/HWS

held_locks / pending_request

operation_cost / agent_cost / blocked_time / retry_count

undo_log_handle / checkpoint_id

不要只用 wid 索引事务。reader list 应存 (tid, generation)，避免事务上下文复用后出现悬空引用。

2. Object Metadata 与锁管理

当前 Python ObjectLockTable 本质上是排他 lease，不能直接等价于论文中的读写锁。需要实现：

ValidateAndRLock(object, tid, observed_version)：在同一元数据临界区完成版本验证和读锁获取。

WLock(object, tid)：处理所有 readers 和 writer owner。

UnlockRead/UnlockWrite。

EnterCommitting/ExitCommitting。

wound 后异步通知事务进入 abort cleanup。

lock queue 删除已 abort、超时或被 wound 的请求。

committing owner 不可被 wound。

多对象锁使用稳定顺序，避免获取一半后形成循环等待。

R_count 必须始终等于有效 R_list 长度，二者更新必须原子。

3. Version Manager

当前接口过窄，需要增加：

ReadCommitted(object_id)

ReadVersion(object_id, version)

CreatePrivateVersion(txn, object_id, value)

ValidateVersion(object_id, expected)

PrepareVersions(txn)

关键点是多对象原子可见性。Undo Log 只能恢复旧值，不能自动保证多个对象同时可见。建议维护 transaction commit table：所有新版本引用 owner_tid，只有事务状态原子变为 COMMITTED 后才可见。

4. Undo Log

当前主代码中没有完整 Undo Log。需要支持以下记录：

BEGIN(txn)

UPDATE(object, old_version, before_image)

CREATE(object)

DELETE(object, old_version, before_image)

VERSION_POINTER(object, old_pointer)

COMMIT(txn)

ABORT(txn)

必须实现：

修改持久数据前先持久化对应 Undo 记录。

中止时按逆序恢复。

启动恢复时识别没有 COMMIT 的事务并回滚。

完成回滚后再释放 committing latch。

checkpoint 和日志截断。

日志 checksum、损坏和重复恢复处理。

大对象使用 block delta、旧版本指针或 copy-on-write，避免保存完整文件副本。

如果暂不实现 crash recovery，接口也应保留，但论文只能声称 logical abort recovery。

5. Unified Commit Protocol

现有主要处理候选分支选择，不应承担底层事务原子性。应将 commit 拆成：

SelectCandidate

AcquireRemainingWriteLocks

ValidateOptimisticReads

SetAllCommittingLatches

AppendAndFlushUndoRecords

InstallPrivateOrBufferedWrites

PublishCommitState

ReleaseLatchesAndLocks

NotifyAgentRuntime

任一步失败都进入统一 AbortAndUndo，不能由不同 CC plugin 各自实现清理。

6. 重构 Pluggable CC 接口

当前只有 Resolve(write, current)，不足以支持运行时切换。建议增加：

OnBegin(ctx)

BeforeRead(ctx, object)

AfterRead(ctx, object, version)

BeforeWrite(ctx, object)

OnPhaseChange(ctx, phase)

OnActionChange(ctx, old_action, new_action)

BeforeCommit(ctx)

OnAbort(ctx)

OnFinish(ctx)

架构原则应是：plugin 负责选择 action 和锁范围，统一 Lock Manager、Version Manager、Commit Protocol 负责安全性。任何 plugin 都不能绕过统一提交协议。

P1：自适应机制

需要补充以下模块：

StateCollector：增量维护 phase、集合增长、hot ratio、abort、lock wait、queue depth、tail latency。

PolicyEngine：加载离线策略表，输出事务级五种 action。

ActionTransition：计算新增保护集合，执行 retroactive validate-and-lock。

PriorityManager：计算 DataOpCost + AgentCost + Retry + Blocked。

HotnessTracker：使用访问率、冲突率、validation failure 和 lock wait 判断热点，不把 YCSB/TPCC key 规则写入核心代码。

PolicyVersionManager：策略版本、原子切换、回滚和兼容性检查。

当前 可以保留为策略层，但需要把 workload-specific hot-key 判断移出核心模块。

P2：系统闭环

还需要：

Agent checkpoint manager：保存可重放的 reasoning state、observations、candidate plans。

Abort notification：数据库事务 abort 后使基于旧数据的 agent state 失效。

Operation trace：记录操作、版本、阶段、action、token cost、锁等待和 Undo LSN。

Backend adapters：关系数据、文本、文件、向量存储分别实现统一事务接口。

Metadata GC：清理 transaction context、reader list、旧版本和已完成 Undo Log。

Monitoring：输出 commit breakdown、lock queue、wound、replay、Undo bytes 等指标。
